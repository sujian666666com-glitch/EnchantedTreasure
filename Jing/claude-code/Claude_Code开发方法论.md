# 用 Claude Code 做开发：实战方法论

> 基于「领星ERP采购对账自动化」项目 33 个 Session（2026-04-16 ~ 04-24）的复盘总结
> 目的：作为下次启动项目时的操作指南，让开发过程更顺畅、产出更可靠

---

## 一、项目启动：建立「AI上下文基础设施」

### 核心原则：AI 没有记忆，你的文档就是它的记忆

每个新 Session，Claude 都是白纸。**上下文恢复的质量直接决定了 AI 的工作质量**。

### 建什么

| 文档 | 作用 | 维护时机 |
|------|------|---------|
| 项目进度总结 | 整体状态、API对接情况、已知问题 | 每次重大改动后更新 |
| AI_review.md | 按Session记录推理过程、决策、踩坑 | 每次Session结束时记录 |
| 关键文件索引 | 哪个文件干什么，快速定位 | 新增/删除文件时更新 |
| CLAUDE.md（项目根目录） | 给Claude的常驻指令（编码规范、偏好） | 一次性设置 |
| MEMORY.md（自动记忆） | 跨会话持久化的关键信息 | 稳定结论才写入 |

### 怎么用

**每次新Session的第一句话**：

```
读 keyfield/项目进度总结.md + keyfield/AI_review.md（重点看最后3个Session）
```

这句指令的目的是：用最少的Token让AI恢复到与上次Session结束时相同的知识状态。之所以说"最后3个Session"而不是全部，是因为：
1. 上下文窗口有限，全读会浪费
2. 最近3个Session通常覆盖了当前工作上下文
3. 更早的历史在项目进度总结中已经有压缩摘要

### 你从这个项目得到的教训

Session 1 做的第一件事就是"读文档恢复上下文"，这个习惯贯穿了33个Session。对比那些"上来就让AI直接写代码"的情况，**先花2分钟读文档，后面能省2小时排错**。

## 二、需求沟通：先验证再动手

### 核心原则：AI 最擅长的是"执行明确的指令"，不是"理解模糊的意图"

### 方法论：「分步确认法」

你的项目中有几次关键的"先沟通后执行"，每次都避免了大量返工：

**案例1：对账状态字段改造（Session 19→20）**

Session 19 只讨论方案，不写代码。产出了一份 `对账状态字段改造方案.md`，明确了：
- 采购单筛选条件是什么
- 哪些旧逻辑要删
- 哪些新逻辑要加
- 哪些边界case怎么处理

Session 20 才按方案执行。结果：一气呵成，核心改造+测试验证一步到位。

**案例2：需求方对接（Session 25）**

先整理了5项需求确认问题，等需求方回复后再设计SQLite迁移方案。如果提前动手，很可能方向就错了。

### 实操建议

```
# ❌ 错误方式
"帮我把对账逻辑改一下，用新的字段来筛选"

# ✅ 正确方式
"我有一个需求想改造，先别动手，帮我分析一下方案"
→ [讨论完方案]
→ "方案确认，按这个文档执行：keyfield/xxx方案.md"
```

### 判断什么时候需要"先讨论"

| 特征 | 直接动手 | 先讨论方案 |
|------|---------|-----------|
| 改动范围 | 1-2个文件 | 3个以上文件 |
| 逻辑确定性 | 明确知道怎么改 | 有多种实现路径 |
| 影响面 | 局部修改 | 涉及核心业务流程 |
| 可逆性 | 容易回滚 | 改了很难回头 |

## 三、验证策略：每个里程碑都要验证

### 核心原则：不要相信代码"应该没问题"，跑一遍才知道

### 你的项目中的验证层次

```
第一层：API对接验证
  → 每个API单独测试，打印原始返回数据
  → 典型错误：假设API文档是对的（实际经常不一致）

第二层：业务逻辑验证
  → 用已知数据（手工模版）对比脚本输出
  → 典型错误：假设计算逻辑没问题（Session 4 第一次验证时误把合计行也加了）

第三层：端到端验证
  → 从API拉取 → 整合 → 对账 → 输出Excel → 人工核对
  → 这是最接近真实使用的验证

第四层：部署验证
  → 本地OK不代表服务器OK
  → Session 21/30 部署时都遇到了环境差异问题
```

### 实操建议

**每个Phase完成后立即验证，不要攒到最后**：

你的Session 26-28（SQLite迁移Phase 1~6）就是好例子：
- Phase 1 完成后：跑迁移脚本，确认237条数据正确导入
- Phase 2 完成后：打开供应商页CRUD操作一遍
- Phase 3 完成后：扣款三Tab各添加/编辑/删除一条
- Phase 5 完成后：跑一次完整对账，确认扣款出现在Excel中

**如果Phase 1~6 全做完再验证，排错时根本不知道问题出在哪一层。**

### 验证的黄金法则

```
能用真实数据验证的，绝不用mock数据
能用完整流程验证的，绝不用单元测试替代
能让人工核对的，绝不相信脚本输出
```

你的项目中最有价值的验证是 Session 4：**拿手工Excel模版逐SKU对比脚本输出**。这个动作一次性确认了计算逻辑的正确性，后面的所有改动都建立在这个可信基础上。

## 四、排错方法论：从现象到根因

### 核心原则：AI 排错的瓶颈不是"不会修"，是"不知道哪里坏了"

### 你的项目中的经典排错案例

**案例1：签名双重编码（Session 3）——间歇性Bug**

| 现象 | 错误假设 | 根因 |
|------|---------|------|
| API调用有时成功有时签名错误 | 是Session的问题 | base64中的`+`被手动URL编码后，requests又编码了一次 |

关键转折点：第4轮测试时打印了签名参数，观察到`%2B`（URL编码的`+`），才意识到问题。

**方法论**：当Bug间歇性出现时，重点关注数据中是否包含特殊字符。纯ASCII数据不会触发编码问题，但base64输出中`+`和`/`的出现是随机的——这正是"间歇性"的来源。

**案例2：1级整合表数据为空（Session 14）——硬编码陷阱**

| 现象 | 排查方向 | 根因 |
|------|---------|------|
| Excel中1级整合表只有表头无数据 | 以为是整合逻辑的Bug | `level1_data=[]` 硬编码为空列表 |

关键转折点：不看逻辑代码，直接看调用方的传参。

**方法论**：当"数据为空"但代码逻辑看起来没问题时，检查调用方是否传了正确的参数，而不是去改被调用函数。

**案例3：扣款不出现在结算Excel（Session 29）——日期格式**

| 现象 | 排查方向 | 根因 |
|------|---------|------|
| 扣款数据在数据库中但不出现在Excel | 以为是reconciler逻辑问题 | SQLite `strftime('%Y','2026/3/7')` 返回NULL |

**方法论**：数据存在但查询不到，首先怀疑查询条件的数据格式是否一致，而不是怀疑查询逻辑。

### 通用排错流程

```
Step 1: 复现问题（确定现象）
  → 打印原始数据，看AI看到的是什么

Step 2: 隔离问题（缩小范围）
  → 是数据问题？逻辑问题？还是传递问题？
  → 方法：在每一层的输入输出打印数据，找到"正确→错误"的转折点

Step 3: 定位根因（找到原因）
  → 不要假设，用数据说话
  → 打开原始数据/API返回值/数据库记录，逐条对比

Step 4: 修复并验证
  → 修复后用同样的测试用例验证
  → 同时检查是否有同类问题（本项目每发现一个扣款去重问题，就检查了其他两种扣款类型）
```

### 给AI的排错指令模板

```
"我发现 [现象]。请帮我排查：
1. 先读 [相关文件]
2. 打印 [关键变量] 的值
3. 逐步分析从 [输入] 到 [输出] 的数据流
4. 找到数据从正确变成错误的转折点"
```

## 五、架构决策：让AI给你方案，你做选择

### 核心原则：AI 擅长列举方案和分析利弊，最终决策由你做

### 你的项目中的架构决策

| 决策点 | AI的角色 | 你的角色 |
|--------|---------|---------|
| 结算基准选quantity_entry还是quantity_real | 分析两个字段的含义 | 确认"按入库量结算"的业务规则 |
| 持久化用JSON还是SQLite | 列举利弊+设计6-Phase方案 | 决定迁移到SQLite |
| 前端用Vue CDN还是构建工具 | 分析项目规模适合CDN模式 | 确认用CDN快速开发 |
| Tkinter还是Web | 分析Web版更适合长期维护 | 确认转FastAPI |

### 实操建议

```
# ✅ 好的提问方式
"我要做一个XX功能的页面，有A和B两种方案，
A的优点是xxx缺点是xxx，B的优点是xxx缺点是xxx，
你觉得哪个更适合？为什么？"

# ❌ 差的提问方式
"帮我做一个页面"
```

**关键**：AI 给方案时，你要追问：
1. "这个方案有什么风险？"
2. "如果后面需求变了，改起来难不难？"
3. "有没有更简单的方案？"

你的项目中有个好例子：Session 25 讨论数据持久化方案时，AI 列举了JSON vs SQLite，分析了项目当前痛点（JSON文件多、数据一致性差、查询能力弱），你基于分析做出了迁移SQLite的决策。这次决策在后续Session中证明非常正确——CRUD操作、按月筛选、去重等需求在SQLite上实现起来比JSON简单得多。

## 六、增量开发：分Phase推进

### 核心原则：大改动拆成小步骤，每步可验证

### 你的项目中的Phase拆分案例

**最佳案例：SQLite迁移（6个Phase）**

```
Phase 1: 数据库层 + 数据迁移        → 验证：跑迁移脚本，数据正确
Phase 2: 供应商页面重写              → 验证：CRUD操作正常
Phase 3: 扣款管理页面（新增）         → 验证：三Tab各操作一遍
Phase 4: 导航栏更新 + 路由注册        → 验证：三页切换正常
Phase 5: reconciler数据源切换        → 验证：跑一次完整对账
Phase 6: 清理旧代码                  → 验证：确认无引用报错
```

**为什么这个拆分好**：
1. 每个Phase独立可验证
2. Phase之间有明确的依赖关系（4依赖3，5依赖1）
3. 即使在某个Phase中断，已完成的部分也是可用的
4. 出问题时范围锁定在一个Phase内

### 反面案例：一次性大改造

Session 10（FastAPI Web版实现）一次性新增了12个文件。虽然最终成功，但如果有问题，排查范围是12个文件，不像Phase拆分那样能快速定位。

### 实操建议

```
# 当AI要一次改很多文件时：
"这个改动太大了，帮我拆成几个Phase，
每个Phase完成后我可以验证一下，
确认没问题再进入下一个Phase"
```

## 七、文档管理：Session间信息传递

### 核心原则：每个Session结束时的文档更新 = 下个Session的启动效率

### 你的项目中的文档体系

```
AI_review.md          → Session级别的详细记录
                        记录：推理过程、决策、踩坑、修改的文件
                        用途：下个Session恢复上下文

项目进度总结.md        → 项目级别的全局状态
                        记录：API对接情况、已完成验证、已知问题、关键文件索引
                        用途：新Session快速了解项目全貌

MEMORY.md（自动）     → 跨项目记忆
                        记录：架构、偏好、踩坑记录
                        用途：自动注入每个Session的上下文
```

### 什么值得记录

| 记录 | 举例 | 不记录 |
|------|------|--------|
| 关键决策及理由 | "用quantity_entry而非quantity_real" | 临时的调试过程 |
| 踩坑和根因 | "双重URL编码导致签名失败" | 修复中间的多次尝试 |
| API返回的真实格式 | "Token接口code='200'，业务接口code=0" | 代码中已有的注释 |
| 业务规则确认 | "入库在结算月 AND 订货不在结算月才参与结算" | 显而易见的逻辑 |
| 文件路径和功能 | "reconciler.py = 对账核心逻辑" | 每次都变的内容 |

### Session结束时给AI的指令

```
"帮我更新以下文档：
1. keyfield/AI_review.md — 新增本次Session的记录
2. keyfield/项目进度总结.md — 更新已完成/待办项"
```

## 八、与需求方的协作模式

### 核心原则：你是AI和需求方之间的翻译器

### 你的项目中的协作流程

```
需求方（采购员/财务）
    ↓ 提供手工Excel模版、描述业务规则
你（开发者+AI操作者）
    ↓ 翻译为技术需求，让AI实现
AI（Claude Code）
    ↓ 生成代码，输出验证数据
你
    ↓ 核对结果，确认或打回
需求方
    ↓ 验收，提出下一轮需求
```

### 你在这个项目中的关键翻译时刻

| 需求方的原话 | 你的翻译 | AI的理解 |
|-------------|---------|---------|
| "按入库量结算" | quantity_entry字段，status=9 | 明确的筛选和计算规则 |
| "扣款要参与结算" | 扣款金额从总金额中减去，按店铺匹配 | net_amount = total - deductions |
| "采购员标记待对账" | custom_fields中对账状态=待对账 | fetcher.py的过滤条件 |
| "送货单NO按规则生成" | 供应商编号-店铺+最晚送货日期-对账月份 | 具体的字符串拼接规则 |

**关键洞察**：需求方说的都是"业务语言"（怎么算钱、怎么对账），AI需要的是"技术语言"（用什么字段、什么条件、什么公式）。你的核心价值就是做这个翻译。

### 实操建议

**拿到需求后的第一步不是让AI写代码，而是**：
1. 把需求翻译成技术要点（字段、条件、规则）
2. 列出你不确定的点
3. 回去问需求方确认
4. 确认后再让AI动手

## 九、部署和运维：从开发到生产

### 核心原则：本地跑通 ≠ 生产可用

### 你的项目中的部署经验

| 经验 | 详情 |
|------|------|
| 部署要早做 | Session 21就开始部署，不是等项目做完 |
| config.yaml不入git | 敏感配置用scp手动传 |
| systemd服务管理 | 开机自启+崩溃自动重启+日志查看 |
| git pull + restart 的更新流 | 简单有效，不需要CI/CD |

### 部署检查清单

```
□ 代码已推送到GitHub
□ 服务器git pull拉取最新
□ config.yaml已手动上传（不在git中）
□ sudo systemctl restart reconcile
□ sudo systemctl status reconcile 确认running
□ 浏览器访问确认页面正常
□ 浏览器强制刷新（Ctrl+Shift+R）确认前端更新
```

### CSS/JS缓存问题

Session 18 踩的坑：修改了前端代码但浏览器看不到更新。

**解决**：每次修改CSS/JS后加版本号参数 `style.css?v=20260421`。这个坑只需要踩一次，后面成为固定做法。

## 附：33个Session的阶段性复盘

### 第一阶段：验证可行性（Session 1-5）

**做了什么**：读文档→修Token→修签名→验证API→修分页
**核心成果**：4个API全部跑通，签名算法修正
**方法论启示**：API对接时，**先打印原始返回数据再写业务逻辑**。你的项目中至少3个Bug是因为假设了API返回格式而未实际验证。

### 第二阶段：业务逻辑搭建（Session 6-9）

**做了什么**：补建单+调整单+入库过滤→硬过滤重构→供应商手动补全
**核心成果**：完整的对账计算逻辑，与手工模版对比验证通过
**方法论启示**：**拿已知答案验证**（手工Excel模版）比写单元测试更有效。复杂的业务逻辑，先让AI实现，然后用真实数据验证。

### 第三阶段：Web化（Session 10-15）

**做了什么**：FastAPI Web版→供应商管理优化→C1-C7整体优化
**核心成果**：桌面版转Web版，7项优化全部完成
**方法论启示**：**平台迁移要一步到位**。从Tkinter到FastAPI不是渐进的，是一次重写。拖泥带水会两套系统并存。

### 第四阶段：业务对齐（Session 16-24）

**做了什么**：金额核对→对账状态改造→API分页优化→服务器部署→退货运费导入→登录鉴权
**核心成果**：脚本输出与手工模版精确对齐，服务器上线
**方法论启示**：**9秒完成对账 vs 手工几小时**——这个效率提升是AI辅助开发的典型成果。关键是业务规则要准确。

### 第五阶段：架构升级（Session 25-33）

**做了什么**：需求对接→SQLite迁移6个Phase→部署→清理→去重优化
**核心成果**：数据层从JSON迁移到SQLite，代码库清理完毕
**方法论启示**：**Phase拆分+逐步验证**是处理大改造的最佳方式。6个Phase跨4个Session，没有出现"改到一半发现方向错了"的情况。

---

## 总结：用Claude Code做开发的5条黄金法则

1. **先恢复上下文，再开始工作** — 每次新Session读文档
2. **先讨论方案，再写代码** — 大改动必有方案文档
3. **每步都验证，不要攒到最后** — Phase拆分+逐步验证
4. **用真实数据验证，不用mock** — 手工模版/API返回值是最好的测试用例
