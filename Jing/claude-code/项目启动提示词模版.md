# 项目启动文档体系 - 提示词模版

> 用途：新项目启动时，用此模版让 AI 自动生成并维护一套标准文档体系
> 提炼自：领星ERP采购对账自动化项目（project-3，33个Session实战经验）

---

## 一、使用方法

### 前置步骤：创建 CLAUDE.md

在项目根目录创建 `CLAUDE.md`（Claude Code 每次启动自动读取），写入项目级常驻指令：

```markdown
# 项目：xxx

## 编码规范
- [你的语言/框架规范，如：Python 3.9+ 兼容，用 Optional 不用 X | None]
- [如：中文注释，变量名英文]

## 常用命令
- 启动：`uv run python main.py --web`
- 测试：`pytest tests/`

## 文档体系
- 项目进度：keyfield/项目进度总结.md
- 决策记录：keyfield/AI_review.md
- 恢复上下文：读上面两个文件（重点看最后3个Session）

## 工作习惯
- 大改动（3+文件）先讨论方案再动手
- 每个 Phase 完成后立即验证
- Session 结束时更新项目文档
```

> **为什么需要 CLAUDE.md**：它是 Claude Code 的"肌肉记忆"。MEMORY.md 是跨项目的，CLAUDE.md 是项目级的。前者存踩坑记录和偏好，后者存编码规范和常用命令。两者自动注入每次对话，不需要你手动指定。

## 二、初始化提示词（首次启动）

```
我要启动一个新项目，请帮我建立项目文档体系。

【项目信息】
- 项目名称：xxx
- 项目目标：xxx（一句话描述）
- 技术栈：xxx
- 数据源/对接方：xxx（如有）
- 已有参考/模版：xxx（如有手工模版、竞品参考、需求文档，说明位置）

【文档体系要求】
请在 keyfield/ 目录下生成以下文档：

1. **spec.md** - 技术规范文档
   - 项目概述（目标、核心逻辑、数据流）
   - 技术栈选型及理由
   - 目录结构设计
   - 配置文件规范
   - 各模块详细设计（职责、接口定义、关键算法）
   - 分步实施计划（Phase划分，每个Phase独立可验证）
   - 关键实现细节（签名算法、转换逻辑等核心代码片段）
   - 错误处理策略
   - 已知限制和后续优化方向

2. **项目进度总结.md** - 项目整体进度（最核心的文档）
   - 项目目标
   - 核心逻辑验证状态
   - 对接情况（API/数据源，逐个列出状态，含真实返回的格式示例）
   - 关键字段/数据匹配情况（哪些有、哪些缺、哪些可计算）
   - 数据源之间的关联关系
   - 已完成的验证项
   - 已知问题与待确认
   - 脚本开发状态（链接到开发进度文档）
   - 关键文件索引（每个文件的职责一句话说明）
   - 服务器/部署信息（如有）

3. **脚本开发进度.md** - 代码层面进度
   - 运行环境配置（Python版本、依赖、启动方式）
   - 完整目录结构（每个文件的职责说明）
   - 各模块功能与逻辑详解
   - 开发进度表（已完成/待完成，含验证状态）
   - 测试验证结果
   - 已知注意事项

4. **待解决问题清单.md** - 按优先级排序
   - P0（阻断正式运行）
   - P1（数据准确性）
   - P2（数据完整性，不影响核心流程）
   - P3（代码质量，功能性完成后处理）
   - 每个问题包含：现状、影响、解决方式、临时方案（如有）

5. **AI_review.md** - AI推理决策记录
   - 按会话记录（Session + 日期）
   - 每次记录包含：背景、推理过程、决策依据、修改的文件清单、经验总结

先通过提问充分了解项目需求，然后依次生成这些文档。
```

> **关于"已有参考/模版"**：这是从 project-3 最重要的经验中提炼的。你的项目之所以能在 Session 4 就确认计算逻辑正确，是因为有手工Excel模版作为"标准答案"。如果有任何参考材料（手工表、竞品截图、需求文档），一开始就告诉AI，这比写100行需求描述都管用。

## 三、持续会话提示词（每次恢复上下文）

```
读 keyfield/项目进度总结.md + keyfield/AI_review.md（重点看最后3个Session）
```

> **为什么不是"读所有文档"**：
> 1. 上下文窗口有限，全读浪费Token在不变的内容上
> 2. 项目进度总结 = 全局状态压缩包，AI_review = 最近工作细节
> 3. "最后3个Session"覆盖了当前工作上下文，更早的在进度总结中已有摘要
> 4. spec.md 和 脚本开发进度.md 只在需要时再针对性读取

**其他文档按需读取**：
- 需要改代码结构时 → `读 keyfield/脚本开发进度.md`
- 需要查架构设计时 → `读 keyfield/spec.md`
- 需要查具体问题时 → `读 keyfield/待解决问题清单.md`

## 四、文档更新规则提示词

```
本次工作已完成，请帮我更新以下文档：

1. 如果有代码变更 → 更新「脚本开发进度.md」的开发进度表和注意事项
2. 如果发现新问题 → 追加到「待解决问题清单.md」对应优先级
3. 如果有关键进展 → 更新「项目进度总结.md」的验证状态和已知问题
4. 无论如何 → 在「AI_review.md」追加本次会话记录（含：背景、推理过程、修改的文件、经验总结）

更新规则：
- 新增内容用日期标注（如：2026-04-24）
- 已解决问题标记 ✅，不要删除（保留解决过程）
- 保持现有文档格式和风格一致
- 不要重复已有内容，只追加增量
```

> **为什么强调"修改的文件"**：project-3 的 AI_review 早期没记录修改文件清单，导致后面排查"这个Bug是什么时候引入的"时需要重新读代码。加上文件清单后，可以快速定位"这个文件的改动是为了什么"。

## 五、开发过程中的关键提示词

### 5.1 大改动前：先讨论方案

```
我需要实现 [功能/改动]，涉及的文件可能有 [文件列表]。
先别动手写代码，帮我分析：
1. 有几种实现方案
2. 各方案的利弊
3. 你推荐哪个，为什么
4. 有什么边界case需要考虑
```

> **使用时机**：改动3+文件、涉及核心业务逻辑、或你不确定怎么做的时候。project-3 中 Session 19（对账状态改造）纯讨论零代码，Session 20 才执行，结果一气呵成。

### 5.2 分Phase推进

```
按刚才讨论的方案执行，请分Phase推进：
- 每个Phase完成后我会验证，确认OK再进下一个
- 每个Phase告诉我：改了哪些文件、怎么验证
- 按依赖关系排列Phase顺序
```

> **使用时机**：任何涉及4+文件或2+功能模块的改动。project-3 的 SQLite 迁移（6个Phase跨4个Session）是最佳案例——每个Phase独立可验证，出问题范围锁定在一个Phase内。

### 5.3 API/数据对接时

```
对接 [API/数据源]，请先帮我：
1. 调用一次API，打印原始返回数据（完整JSON）
2. 分析实际返回的字段名和类型，与文档对比
3. 确认分页机制（offset/length 还是 page/page_size）
4. 确认日期筛选方式
确认无误后再写业务逻辑
```

> **使用时机**：对接任何外部API或数据源。project-3 中至少4个Bug源于"假设API返回格式与文档一致"——Token接口code是字符串"200"不是整数0、入库单分页用offset/length不是page/page_size、custom_fields的字段名是name不是field_name。

### 5.4 排查Bug

```
我发现 [现象]。请帮我排查：
1. 先读 [相关文件1] 和 [相关文件2]
2. 在 [关键环节] 打印中间数据
3. 逐步分析从 [输入] 到 [输出] 的数据流
4. 找到数据从正确变成错误的转折点
不要急着修复，先定位根因。
```

> **使用时机**：任何"数据不对"的场景。project-3 的经典案例：1级整合表数据为空——不是整合逻辑有问题，是调用方传了空列表。如果一开始就去改整合逻辑，方向完全错了。

### 5.5 验证产出

```
请帮我验证 [功能/改动] 的正确性：
1. 用 [真实数据/已知结果] 对比脚本输出
2. 逐项核对 [关键维度]，列出差异
3. 对差异逐条分析原因
```

> **使用时机**：每个Phase完成后、每次核心逻辑改动后。最佳验证数据 = 需求方提供的手工模版/已知答案。project-3 的 Session 4 用手工Excel逐SKU对比，一次性确认了全部计算逻辑的正确性。

## 六、文档体系设计原则

从 project-3 的实践经验中提炼：

| 原则 | 说明 |
|------|------|
| **分层记录** | spec（不变）> 进度总结（阶段性更新）> 开发进度（频繁更新）> AI_review（每次追加） |
| **快速恢复** | 进度总结文档的核心目的 = 新会话30秒内恢复完整上下文 |
| **问题可追踪** | 待解决问题按优先级排序，每个问题含现状/影响/解决方式 |
| **决策留痕** | AI_review 记录"为什么"而不只是"做了什么"，必须包含修改的文件清单 |
| **相互引用** | 文档之间用链接互引，避免信息重复 |
| **真实数据优先** | 记录API真实返回格式示例，不要只记文档描述 |

### 文档更新频率

```
CLAUDE.md             ── 项目级常驻指令，一次性设置，架构调整时更新
MEMORY.md             ── 跨项目记忆（Claude自动管理），稳定结论才写入
spec.md               ── 初始生成后基本不变，重大架构调整时更新
项目进度总结.md        ── 每个里程碑更新（完成一个Phase、验证通过一个API等）
脚本开发进度.md        ── 每次代码变更后更新
待解决问题清单.md      ── 发现问题立即追加，解决后标记 ✅ 保留
AI_review.md          ── 每次会话追加一条记录（含修改文件清单）
```

### 信息不重复原则

| 信息类型 | 归属文档 | 其他文档的处理 |
|---------|---------|--------------|
| 架构设计、模块接口 | spec.md | 其他文档只引用不展开 |
| API状态、字段验证 | 项目进度总结.md | 开发进度只写"已验证" |
| 代码结构、模块逻辑 | 脚本开发进度.md | 进度总结只写"详见xx文档" |
| 当前问题 | 待解决问题清单.md | 进度总结只写问题数量 |
| 推理决策过程 | AI_review.md | 其他文档只写结论 |
| 编码规范、常用命令 | CLAUDE.md | 不在文档中重复 |
| 踩坑记录、偏好 | MEMORY.md | 不在文档中重复 |

### 文档规模控制

```
项目进度总结.md   ── 控制在 400 行以内（过长说明需要拆分或压缩旧内容）
AI_review.md      ── 不限长度，但每个Session控制在 50 行左右
spec.md           ── 不限，但要结构清晰有目录
```

> **为什么限制进度总结长度**：project-3 的进度总结到400+行时，新Session读取就要消耗大量上下文窗口。定期压缩已稳定的内容（如"已完成的验证"可以合并为一行），保持文档精简。

## 七、适配不同项目的建议

| 项目类型 | 保留 | 简化 | 新增 |
|---------|------|------|------|
| **API对接类**（如本项目） | API状态表、字段匹配表、关联关系图 | — | API真实返回示例 |
| **纯脚本/工具类** | spec中的核心算法 | API相关内容 | 输入输出规范、CLI参数设计 |
| **Web应用类** | spec中的模块设计 | — | 路由设计、数据库Schema、前端组件结构 |
| **数据分析类** | 数据源说明 | API对接细节 | 清洗规则、分析模型定义 |

### 复杂度分级

| 项目规模 | 建议文档 | 可省略 |
|---------|---------|--------|
| **小型**（1-2天，单文件脚本） | 项目进度总结 + AI_review | spec、待解决问题清单、脚本开发进度 |
| **中型**（1-2周，5-10个文件） | 全套文档 | — |
| **大型**（1月+，多模块） | 全套 + 额外拆分（如按模块的spec） | — |

## 附：常见场景的指令速查

| 场景 | 指令 |
|------|------|
| 新Session启动 | `读 keyfield/项目进度总结.md + keyfield/AI_review.md（最后3个Session）` |
| 新功能开发 | `先别动手，帮我分析实现方案` → 讨论完 → `分Phase执行` |
| API对接 | `先调用一次打印原始返回，确认后再写逻辑` |
| Bug排查 | `先定位根因，不要急着修复` |
| 产出验证 | `用[已知数据]对比输出，逐项核对` |
| Session结束 | `更新项目文档（进度+review+问题清单）` |
| 部署更新 | `帮我推代码到GitHub，列出部署步骤` |
| 大改造 | `先写方案文档到keyfield/，我确认后再执行` |
