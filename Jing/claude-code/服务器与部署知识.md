## 一、服务器基础

### 1.1 什么是云服务器

云服务器就是远程的一台电脑，7x24小时运行，通过公网IP从任何地方访问。

| 概念 | 说明 |
|------|------|
| 公网IP | 服务器在互联网上的地址，如 `106.52.244.190` |
| 端口 | 一台机器有65535个端口，每个服务监听不同端口（如Web用80、HTTPS用443） |
| SSH | 远程登录服务器的协议，默认端口22 |
| systemd | Linux的服务管理器，让程序开机自启、崩溃自动重启 |

### 1.2 腾讯云服务器选择

**轻量应用服务器 vs CVM云服务器**：

| 对比 | 轻量应用服务器 | CVM云服务器 |
|------|--------------|------------|
| 定位 | 简单易用，适合小项目 | 灵活可控，适合企业级 |
| 带宽 | 包含固定带宽 | 按量或按带宽计费 |
| 价格 | 便宜（活动价年付） | 较贵（按月付费） |
| 扩容 | 支持升配，不支持降配 | 支持升降配 |
| 适合 | 几个项目、30人以下团队 | 大规模、需要弹性伸缩 |

**30人公司推荐：4核8G 轻量应用服务器**（活动价约 ¥400-600/年）

### 1.3 镜像选择


镜像 = 服务器的预装系统。

| 镜像 | 内容 | 推荐场景 |
|------|------|---------|
| Ubuntu Server 22.04 LTS | 纯净Ubuntu | **通用推荐** |
| Ubuntu + Docker | 预装Docker | 需要容器化部署时 |
| 宝塔面板 | 预装Web管理面板 | 不想敲命令时 |

**建议选纯净 Ubuntu**，需要什么自己装更灵活。

### 1.4 配置可以升级吗

可以。在腾讯云控制台在线操作，停机几分钟自动迁移，数据不丢。先买低配，后续不够再升。

## 二、多项目部署

### 2.1 一台服务器部署多个项目

完全可以。每个项目监听不同端口，用 **Nginx 反向代理** 按域名分流。

```
服务器（公网IP）
  └── Nginx（监听 80/443）
        ├── erp.company.com    → localhost:8000  （对账系统）
        ├── ai.company.com     → localhost:8001  （AI中控）
        └── tool.company.com   → localhost:8002  （其他工具）
```

每个项目各跑各的 systemd 服务，互不干扰。

### 2.2 反向代理原理

**没有反向代理**：用户记 IP+端口号（`http://106.52.244.190:8000`）

**有反向代理**：用户记域名（`erp.xxx.com`），Nginx 在中间转发

```
1. 用户访问 erp.xxx.com
2. DNS 解析 → 服务器IP
3. 请求到 Nginx（80端口）
4. Nginx 看域名是 erp.xxx.com → 转发给 localhost:8000
5. FastAPI 处理完 → 返回 Nginx → 返回用户
```

Nginx 就是门口的"前台"，根据域名把请求转给对应的服务。

### 2.3 域名

| 有没有域名 | 区别 |
|-----------|------|
| 没有 | 用 `http://IP:端口` 访问，记端口号 |
| 有 | 用 `erp.xxx.com` 访问，专业、直观 |

**域名不是必须的，但强烈建议买一个**：
- .com 域名约 ¥55-70/年
- 一个域名可以配无数个子域名（erp.xxx.com、ai.xxx.com、tool.xxx.com）
- 买域名后可以配 HTTPS（安全加密）
- IP变了改DNS就行，不用通知所有人

**可以先没域名用IP跑着，后面随时加，项目代码零改动。**

## 三、Docker 基础

### 3.1 Docker 是什么

Docker 是把应用和它的运行环境打包成一个"集装箱"（镜像），搬到任何服务器都能直接跑。

```
传统方式：
  服务器
    ├── Python 3.9（项目A）
    ├── Node.js 18（项目B）
    └── 依赖可能互相冲突

Docker方式：
  服务器
    └── Docker引擎
          ├── 容器A（自带 Python 3.9 + 依赖 + 代码）
          └── 容器B（自带 Node.js 18 + 依赖 + 代码）
```

### 3.2 核心概念

| 概念 | 类比 |
|------|------|
| **镜像 (Image)** | 安装包，只读模板 |
| **容器 (Container)** | 运行中的程序，从镜像启动 |
| **Dockerfile** | 安装说明书，描述怎么打包 |

### 3.3 Docker 适合的场景

- 团队协作标准化（"我这能跑你那跑不了"）
- 微服务架构（几十个服务独立部署）
- 一键部署开源软件（`docker run wordpress`）
- 大规模服务器管理（配合 Kubernetes）

### 3.4 现阶段需要 Docker 吗

**不需要。**

30人公司、3-4个Python项目，直接跑（`uv run python main.py`）更简单。Docker 多一层抽象，排查问题更麻烦。

等以后项目超过10个、或者需要严格的环境一致性时再考虑。

## 四、部署实践（本项目）

### 4.1 当前部署架构

```
腾讯云轻量服务器（2核2G）
  ├── systemd 服务 reconcile
  │     └── uv run python main.py --web（监听8000端口）
  ├── Nginx（暂未配置）
  └── 访问方式：http://106.52.244.190:8000
```

### 4.2 部署流程

```bash
# 首次部署
git clone https://github.com/gj666m/gj.git /home/ubuntu/project-3
cd /home/ubuntu/project-3
uv sync
vim config.yaml          # 手动创建配置文件
sudo systemctl enable --now reconcile

# 日常更新
cd /home/ubuntu/project-3
git pull
sudo systemctl restart reconcile
```

### 4.3 config.yaml 安全处理

- `config.yaml` 已加入 `.gitignore`，不进版本控制
- 仓库改为私有，通过 GitHub Personal Access Token 认证
- 服务器需手动创建 `config.yaml`（含API凭证 + auth用户）

### 4.4 服务管理命令

```bash
sudo systemctl start reconcile     # 启动
sudo systemctl stop reconcile      # 停止
sudo systemctl restart reconcile   # 重启
sudo systemctl status reconcile    # 查看状态
sudo journalctl -u reconcile -f    # 查看实时日志
```

## 五、日常运维操作手册

> 本节记录项目中常用的服务器操作指令，按使用频率排序。

### 5.1 连接服务器

```bash
ssh ubuntu@106.52.244.190
```

首次连接会提示指纹确认，输入 `yes` 回车即可。密码为服务器密码。

### 5.2 更新代码并重启服务（最常用）

每次本地改完代码推到 GitHub 后，在服务器上执行：

```bash
cd /home/ubuntu/project-3 && git pull && sudo systemctl restart reconcile
```

如果服务器上 `config.yaml` 有本地改动（会被 git pull 冲突）：

```bash
cd /home/ubuntu/project-3
git stash                  # 暂存本地改动（主要是 config.yaml）
git pull                   # 拉取最新代码
git stash pop              # 恢复本地改动
sudo systemctl restart reconcile
```

> 注意：`config.yaml` 已在 `.gitignore` 中，正常不会冲突。但 `data/` 目录下的文件可能有问题，用 `git stash` 最安全。

### 5.3 查看服务状态

```bash
# 查看服务是否在运行
sudo systemctl status reconcile

# 查看实时日志（Ctrl+C 退出）
sudo journalctl -u reconcile -f

# 查看最近50行日志
sudo journalctl -u reconcile -n 50

# 查看今天的日志
sudo journalctl -u reconcile --since today
```

**日志关键信息**：
- 看到 `Uvicorn running on http://0.0.0.0:8000` → 服务启动成功
- 看到 `ERROR` 或 `Traceback` → 有报错，看上下文定位
- 看到 `code=3001008` → API 限流，正常，脚本会自动重试

### 5.4 编辑配置文件

```bash
# 编辑 config.yaml（含API凭证、登录账号等）
vim /home/ubuntu/project-3/config.yaml
```

**vim 基本操作**：
- `i` → 进入编辑模式
- 编辑完成后按 `Esc`
- `:wq` → 保存退出
- `:q!` → 不保存退出

编辑后需重启服务：`sudo systemctl restart reconcile`

### 5.5 修改登录账号密码

config.yaml 中的 `auth.users` 部分控制登录账号：

```yaml
auth:
  users:
    - username: "admin"
      password: "你的密码"
```

修改后重启服务生效。

### 5.6 查看端口和进程

```bash
# 检查8000端口是否在监听
sudo ss -tlnp | grep 8000

# 查看项目进程
ps aux | grep main.py

# 查看服务器内存使用
free -h

# 查看磁盘使用
df -h
```

### 5.8 常见问题排查

| 现象 | 排查命令 | 可能原因 |
|------|---------|---------|
| 页面打不开 | `sudo systemctl status reconcile` | 服务挂了，重启 |
| 页面能打开但对账报错 | `sudo journalctl -u reconcile -n 100` | Token过期或API异常 |
| 对账一直转圈无响应 | `ps aux \| grep main.py` | 进程卡住，`sudo systemctl restart reconcile` |
| git pull 报错冲突 | `git stash && git pull && git stash pop` | 本地文件与远程冲突 |
| 磁盘满 | `df -h` 然后清理 `outputexcel/` 和 `logs/` | 历史输出文件太多 |
| Token 报错 401 | 查看 config.yaml 中 token 是否为空 | 需重新获取 Token |

### 5.9 清理磁盘空间

```bash
# 查看哪些目录占空间大
du -sh /home/ubuntu/project-3/* | sort -rh | head -10

# 清理历史对账输出（谨慎，确认不需要后删）
rm -rf /home/ubuntu/project-3/outputexcel/结算单_*

# 清理日志（保留最近7天）
sudo journalctl --vacuum-time=7d
```
